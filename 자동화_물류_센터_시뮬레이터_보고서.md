자동화 물류 센터 시뮬레이터 개발
20241416 김보현

1. 개요
   본 프로젝트는 중앙 관제 스레드와 여러 개의 로봇 스레드로 자동화 물류 센터를 시뮬레이션한다. 로봇들은 물건을 적재 후 하역 장소에 내려놓는 것을 목표로 한다.

2. 구현 방법
   2.1 파라미터 파싱
   run_automated_warehouse 함수에서 초기 파라미터를 파싱하여 중앙 관제 스레드와 로봇 스레드를 동적으로 생성한다. 예를 들어 `automated_warehouse 3 2A:4C:2B`가 입력되면 로봇 3대를 생성하고, 각각 2번 물건을 A로, 4번 물건을 C로, 2번 물건을 B로 옮기도록 설정한다.
   2.2 스레드의 순서 관리
   스레드의 순서는 wait_steps 배열을 이용해 관리했다. 이동하지 못한 로봇은 wait_steps 값이 증가하고, 이동에 성공한 로봇이나 작업을 완료한 로봇은 wait_steps 값이 0으로 초기화된다. 중앙 관제 스레드는 한 스텝에서 이동 후보를 검사할 때 wait_steps 값이 큰 로봇, 즉 오래 기다린 로봇부터 먼저 이동 가능 여부를 판단한다. 이렇게 하여 특정 로봇이 계속 뒤로 밀리는 상황을 줄이고, 오래 기다린 로봇이 먼저 나갈 기회를 갖도록 하였다.
   2.3 메시지
   중앙 관제 스레드와 로봇 스레드의 통신은 메시지 박스를 매개로 이루어진다. 로봇에서 중앙 관제 노드로 가는 메시지와 중앙 관제 노드에서 로봇으로 가는 메시지를 구분하기 위해 `boxes_from_robots`와 `boxes_from_central_control_node` 배열을 각각 동적 할당하였다. 로봇 스레드는 중앙 관제 스레드에 현재 위치, 현재 적재량, 요구 적재량, 작업 완료 여부 등의 정보를 메시지로 전달한다. 로봇의 이름, 목표 물건, 목적지는 robot 구조체에서 관리한다. 중앙 관제 스레드는 각 로봇에게 내린 명령을 메시지로 전달한다. 메시지가 새로 쓰였는지 확인하기 위해 `dirtyBit`를 사용하였다.
   2.4 각 스텝의 중앙 관제 스레드와 로봇 스레드의 통신
   로봇 스레드는 매 스텝마다 자신의 현재 위치, 현재 적재량, 요구 적재량, 작업 완료 여부를 메시지로 중앙 관제 노드에 전달한다. 로봇의 이름, 운반할 물건, 목적지는 `robot` 구조체에 저장되어 관리된다.
   중앙 관제 스레드는 모든 로봇의 `dirtyBit`가 켜질 때까지 기다린다. 모든 메시지가 도착하면 `receive_message_from_robot()`로 로봇 상태를 읽어 현재 상태를 반영하고, 그 상태를 기준으로 `print_map()`을 호출하여 시뮬레이터 상태를 출력한다.
   중앙 관제 노드가 로봇에게 내리는 명령은 `cmd` 값으로 전달된다. `0`은 대기, `1`은 위, `2`는 아래, `3`은 왼쪽, `4`는 오른쪽 이동을 의미한다. 중앙 관제 노드는 `send_command_to_robot()`를 통해 각 로봇의 메시지 박스에 명령을 기록하고, 로봇은 unblock된 뒤 `receive_command_from_central()`로 그 명령을 읽어 실행한다.
   2.5 block과 unblock
   중앙 관제 스레드는 우선 모든 로봇 스레드가 block인 상태에서 매 스텝을 처리한다. 로봇 스레드에서 온 메시지를 취합하고, 각 로봇의 다음으로 이동하고자 하는 위치를 미리 계산하여, 그 위치로 이동이 가능한지를 판단한다. 먼저 모든 로봇에게 wait 명령을 넣어 두고, 이동이 가능한 로봇 스레드에만 상하좌우 이동 명령을 덮어쓴다. 그 다음 increase_step()으로 단위 스텝을 증가시키고, 마지막으로 모든 로봇 스레드를 unblock한다.
   2.6 인터럽트 관리
   로봇 스레드는 자신의 상태를 중앙 관제 스레드에 전달한 후 block_thread()를 호출하여 대기 상태가 된다. 중앙 관제 스레드는 모든 로봇의 상태를 받은 뒤 이동 명령을 생성하고, unblock_threads()를 호출하여 대기 중인 로봇들을 다시 실행 가능 상태로 만든다. block_thread()와 unblock_threads() 중간에 선점이 발생하는 것을 방지하기 위해 intr_disable()로 인터럽트를 비활성화하고, 작업이 끝나면 intr_set_level()을 이용해 이전 인터럽트 상태로 복구하였다.
   2.7 최적 경로 찾기
   로봇의 이동 경로는 BFS를 이용해 계산하였다. 중앙 관제 스레드는 로봇이 아직 물건을 적재하지 않았다면 해당 물건의 위치를 목표로 하고, 물건을 적재한 상태라면 지정된 하역 장소를 목표로 한다. get_next_direction이 BFS로 이동 방향을 정하는 함수이다.
   2.8 기타
   robot 구조체에 작업 완료 여부를 추가하였다. 이를 통해 중앙 관제 스레드는 모든 로봇이 작업을 완료했는지 확인할 수 있고, 완료된 로봇은 더 이상 이동 후보로 선택하지 않는다.

3. 트러블 슈팅
   3.1 초기에는 한 스텝에 하나의 로봇만 이동시키는 방식으로 구현하려고 하였으나, 이렇게 하면 복잡한 상황에서는 스텝이 너무 많아지고, 멀티스레딩 시스템을 충분히 이용하지 못한다는 단점이 있었다. 따라서 한 스텝에서 현재 위치, 이동하고자 하는 위치들이 서로 겹치지 않는 로봇들에 한해 동시에 움직이게 하여 멀티스레딩을 구현하였다. 스레드들의 위치가 겹치는지의 여부는 can_accept_batch_move 함수로 체크한다.
   이때 단순히 다음 위치만 비교하면 두 로봇이 서로 자리를 맞바꾸는 경우를 놓칠 수 있다. 따라서 can_accept_batch_move 함수에서는 이동하려는 로봇들의 current와 next를 모두 비교하여, 현재 위치와 다음 위치가 하나라도 겹치면 동시에 이동시키지 않도록 하였다.
   3.2 충돌한 로봇들이 서로 반대 방향으로 가고자 하여 정체 구간이 생기는 ‘데드락’ 현상이 발생하면 기존의 로직에서는 무한 wait 상황이 발생한다. 따라서 데드락 현상이 발생하면, 탈출을 위해 한 로봇을 victim으로 삼아 목표 지점에서 멀어지는 방향으로 이동하도록 한다. 다른 로봇이 정체 구간을 탈출하면 다시 원래의 모드로 돌아간다. victim을 정하는 함수는 choose_deadlock_victim이고, 현재 escape가 가능한 로봇 중 wait_steps 값이 큰 로봇을 우선 선택한다. get_escape_direction 함수는 victim 로봇이 목표 지점에서 멀어지는 방향으로 이동하도록 한다.
